WSL2 Setup Guide

FF Office Hours — Run a full Linux environment inside Windows

Do you need WSL2?

WSL2 (Windows Subsystem for Linux 2) gives you a real Linux environment running inside Windows. It's not required to use any of the AI CLI tools — Claude Code, Gemini CLI, and Codex all work natively on Windows with PowerShell. WSL2 is an optional upgrade for people who want a more developer-friendly environment.

Here's a quick way to decide:

Install WSL2 if you...

  • Plan to write Python scripts or automation tools
  • Want to use Docker for containers
  • Work with repos that use bash scripts or Makefiles
  • Prefer Linux command-line tools (grep, sed, awk)
  • Want an environment that matches most online tutorials
  • Are comfortable with (or want to learn) the terminal

Skip WSL2 if you...

  • Just want to try the AI CLI tools quickly
  • Are brand new to the command line
  • Only plan to use these tools casually
  • Don't want to manage two environments
  • Have a Mac (macOS is already Unix-based)
  • Prefer to keep things simple for now
1

Install WSL2 & Ubuntu

The prerequisite for everything else: a running Linux environment. We'll keep this phase minimal — just enough to get Ubuntu booting so we can install an AI CLI inside it.

Requirements

Check your Windows version

WSL2 requires Windows 10 version 1903 or later, or Windows 11. To check your version, press Win + R, type winver, and hit Enter.

Install

Run the WSL installer

Open PowerShell as Administrator (right-click the Start button → "Terminal (Admin)" or search for PowerShell and right-click → "Run as administrator"):

wsl --install

This enables WSL2 and installs Ubuntu in one shot. Restart your computer when prompted.

Want a different distro? Run wsl --list --online to see available distros, then wsl --install -d Debian (or whichever you prefer). Ubuntu is the most beginner-friendly and is assumed throughout the rest of this guide.
Setup

Create your Linux user

After your computer restarts, Ubuntu will launch automatically (or search "Ubuntu" in the Start menu). It will ask you to create a username and password:

# You'll see something like this:
Enter new UNIX username: yourname
New password: ********
Retype new password: ********
Remember this password! You'll need it whenever you run sudo commands. It doesn't have to match your Windows password.
Update

Update system packages

Always update packages right after a fresh install:

sudo apt update && sudo apt upgrade -y
2

Install an AI CLI (pick at least one)

You need at least one AI CLI running inside WSL2 so Phase 3 can be automated. Pick whichever you have an account for — you can add the others later from Phase 3. All three can coexist.

Pick & Install

Choose your AI CLI

Click a tab to see the install steps for that CLI. Claude Code has no dependencies; Gemini and Codex both need Node.js, which each tab includes.

Account required: Claude Pro or Max ($20+/mo). Free Claude.ai accounts aren't eligible.

# Install — native binary, no Node.js needed
curl -fsSL https://claude.ai/install.sh | bash

# Verify
claude --version

# Authenticate (browser opens automatically)
claude

Account required: Google account (free tier gives you 60 req/min and 1,000 req/day).

# 1. Install Node.js 20+ via nvm
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.1/install.sh | bash
source ~/.bashrc
nvm install --lts

# 2. Install Gemini CLI
npm install -g @google/gemini-cli

# 3. Authenticate — select "Sign in with Google"
gemini

Account required: ChatGPT Plus, Pro, Team, or Enterprise.

# 1. Install Node.js 20+ via nvm (skip if already installed)
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.1/install.sh | bash
source ~/.bashrc
nvm install --lts

# 2. Install Codex
npm install -g @openai/codex

# 3. Authenticate — select "Sign in with ChatGPT"
codex
API key works too: set OPENAI_API_KEY in your environment if you'd rather use API billing instead of a ChatGPT subscription.
Browser links work from WSL2! When any of these CLIs says "open this URL to authenticate," it will automatically open your Windows browser even though you're inside Linux.
3

Use your AI CLI to finish setup

Now that you have a working AI CLI, let it handle the rest: Python, Git config, the AI CLIs you haven't installed yet, and a PowerShell shortcut so you can launch Ubuntu by typing ubuntu. Works with Claude Code, Gemini CLI, or Codex — the prompt below is the same for all three.

Prompt

Ask your CLI to set up your environment

From your WSL2 terminal, launch whichever CLI you installed (claude, gemini, or codex) and paste something like this:

# Example prompt:
I'm on a fresh WSL2 Ubuntu. Please set up my dev environment step
by step, showing me each command before running it. First check
what's already installed and skip anything that's there.

1. Install Node.js 20+ via nvm (if not already installed)
2. Install Python 3, pip, and venv via apt
3. Install any of these AI CLIs that aren't already present:
   Claude Code (curl https://claude.ai/install.sh | bash),
   Gemini CLI (npm i -g @google/gemini-cli),
   Codex (npm i -g @openai/codex)
4. Configure git with my name and GitHub no-reply email (ask me for
   these)
5. Add a function to my Windows PowerShell $PROFILE so typing
   `ubuntu` from any PowerShell prompt launches WSL Ubuntu. My
   Windows username is <YOURNAME>.
Why "show me each command before running it"? All three CLIs will ask permission for system-modifying commands by default, but being explicit keeps you in a review-and-approve flow — you see exactly what's being changed and learn the underlying commands as you go.
SSH keys for GitHub: Once the above finishes, ask your CLI to walk you through the "Linux / WSL2" SSH section of the Git Setup Guide. Your Windows SSH keys don't carry over — you need fresh ones inside WSL2.
Manual Fallback

Prefer to run the commands yourself?

If you'd rather skip the AI CLI for this part, the raw commands are below. Run them in your WSL2 terminal in order.

# 1. Node.js via nvm
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.1/install.sh | bash
source ~/.bashrc
nvm install --lts

# 2. Python toolchain
sudo apt install python3 python3-pip python3-venv -y

# 3. Other AI CLIs
npm install -g @google/gemini-cli
npm install -g @openai/codex

# 4. Git identity (use your GitHub no-reply email)
git config --global user.name "Your Name"
git config --global user.email "ID+username@users.noreply.github.com"
git config --global init.defaultBranch main

For the PowerShell ubuntu shortcut, open PowerShell on Windows and run notepad $PROFILE (say yes if it offers to create the file), then add and save:

function ubuntu { wsl -d Ubuntu }

See our Git Setup Guide for full SSH-key instructions (use the "Linux / WSL2" tab for each step).

4

Working with Files

Access

Access your Windows files from WSL2

Your Windows drives are automatically mounted under /mnt/:

# Your Windows home folder
cd /mnt/c/Users/YourWindowsUsername

# Desktop
cd /mnt/c/Users/YourWindowsUsername/Desktop

# A project on your D: drive
cd /mnt/d/projects/my-project
Performance

Where to keep your projects

For best performance, keep project files inside the WSL2 filesystem rather than on your Windows drives:

# Create a projects folder in your WSL home (fast)
mkdir ~/repos
cd ~/repos
git clone git@github.com:username/repo.git

# AVOID working in /mnt/c/ for large projects (slow)
Why? Reading and writing files across the Windows ↔ WSL boundary is significantly slower than working within the WSL filesystem. For small scripts it won't matter, but for projects with many files (like node_modules) the difference is huge.
Explorer

Open WSL files in Windows

You can browse your WSL files from Windows Explorer:

# From inside WSL, open the current folder in Windows Explorer
explorer.exe .

# Or type this in Windows Explorer's address bar:
# \\wsl$\Ubuntu\home\yourname\repos
5

VS Code & WSL2

The "Remote - WSL" workflow is the gold standard for Windows development. It allows you to run VS Code on Windows while the actual code, tools, and terminal run inside Linux.

Install

1. Install VS Code on Windows

If you haven't already, download and install Visual Studio Code on Windows. Do not try to install the Linux version of VS Code inside WSL.

Extension

2. Install the WSL Extension

Open VS Code on Windows, go to the Extensions view (Ctrl+Shift+X), and search for "WSL". Install the official extension by Microsoft.

Launch

3. Open a project from WSL

Now, go back to your WSL Ubuntu terminal, navigate to a project folder, and type:

code .

VS Code will open on Windows, but it will be "connected" to your Ubuntu environment. You'll see WSL: Ubuntu in the bottom-left corner.

Extensions in WSL: Some extensions (like Python or ESLint) need to be installed "specifically" in WSL. When you open a project, check the Extensions view — you'll see a section titled "WSL: Ubuntu - Installed" where you can add them.

WSL2 Quick Reference

# Open WSL from PowerShell
wsl

# Open WSL in the current Windows directory
wsl --cd .

# Open VS Code from inside WSL
code .

# Open Windows Explorer from inside WSL
explorer.exe .

# Run a single Linux command from PowerShell
wsl ls -la

# Check which distros are installed
wsl --list --verbose

# Shut down WSL (frees RAM)
wsl --shutdown

# Check WSL version
wsl --version

# Open a specific distro
wsl -d Ubuntu

Troubleshooting

"WSL 2 requires an update to its kernel component"
Download and install the WSL2 kernel update from aka.ms/wsl2kernel, then restart.
"Please enable the Virtual Machine Platform Windows feature"
Virtualization needs to be enabled. Run in admin PowerShell: 
dism.exe /online /enable-feature /featurename:VirtualMachinePlatform /all /norestart
Then restart your computer. If it still doesn't work, you may need to enable virtualization (VT-x or AMD-V) in your BIOS settings.
WSL is very slow when accessing /mnt/c/ files
This is expected — cross-filesystem I/O is slow by design. Move your projects into the WSL filesystem (~/repos/) for much better performance. Only use /mnt/c/ when you need to access files that must stay on Windows.
Git/SSH keys from Windows don't work in WSL2
WSL2 has its own filesystem — your Windows C:\Users\You\.ssh keys aren't automatically available inside WSL. You need to either generate new keys inside WSL (recommended) or copy them: 
cp /mnt/c/Users/YourName/.ssh/id_ed25519* ~/.ssh/
chmod 600 ~/.ssh/id_ed25519
chmod 644 ~/.ssh/id_ed25519.pub
"npm WARN EBADPLATFORM" or packages won't install
If you installed Node.js via the Windows installer, it won't work inside WSL. Install Node.js separately inside WSL using nvm (see Phase 3 above — either ask Claude or run the manual commands). Windows and WSL have completely separate software installations.
WSL is using too much RAM
WSL2 can be greedy with memory. Create a .wslconfig file in your Windows home folder to set limits: 
# Create from PowerShell:
notepad $env:USERPROFILE\.wslconfig
Add these contents and save: 
[wsl2]
memory=4GB
processors=2
Then restart WSL: wsl --shutdown
